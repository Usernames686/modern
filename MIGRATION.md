# 业务迁移清单

当前交付口径以 `modern/DELIVERY_GOAL.md` 为准：功能对齐、体验优化、快速交付；参考官方 `crmeb/crmeb_java`、`crmeb/CRMEB` 与本地老项目，但必须按模块移植或旧版兼容保留，不能整仓覆盖已跑通的 modern 工程。

## 总原则：功能级替代

- 新技术栈替换底层工程、依赖、构建和部署方式，业务能力、接口路径、数据结构和核心流程必须对齐老系统。

- 用户端必须保留老项目的页面路径、业务流程、图片展示、购物车/下单/支付等完整能力；UI 可以在新版 H5 中重新整理，但不能减少功能入口。
- 运行代码旧 IP 扫描已收口：`app/config/app.js`、`modern/admin-web/src`、`modern/app-web/src`、`modern/app-web/scripts`、`modern/backend`、`modern/README.md`、`modern/MIGRATION.md`、`modern/H5_STACK_UPGRADE.md`、`modern/DELIVERY_GOAL.md` 中不再出现旧项目硬编码域名或内网 IP 命中。

## 每个模块的迁移验收

- API 路径与老系统一致。
- 数据库表不改名、不删字段。
- 老系统同一接口的关键返回字段在新接口中仍存在。
- 页面 UI 与老系统同模块保持一致，包括布局、入口、按钮文案、筛选项、表格列、弹窗和移动端适配。
- 业务功能与老系统同模块保持一致，不能缺少新增、编辑、删除、导入导出、状态切换等原有操作。
- 后台和用户端页面能通过 `/api` 代理正常调用。
- 单模块迁移后不影响未迁移模块继续走老系统或后续迁移。
